# STM32 Mill 第二阶段本地升级手册

本文档用于说明第二阶段“STM32 本地升级”的实际操作方法、Bootloader 协议格式和建议联调步骤。

本文档面向：

- 现场维护人员
- 上位机/脚本对接人员
- Bootloader 联调人员

## 1. 适用范围

适用对象：

- MCU：`STM32F103ZET6 / STM32F103ZE`
- 本地升级物理通道：`USART3 + 485`
- App 运行态维护协议：`Modbus RTU`
- Bootloader 升级协议：自定义二进制帧协议

## 2. 当前分区

| 区域 | 地址范围 | 大小 | 说明 |
|---|---|---:|---|
| Bootloader | `0x08000000 ~ 0x08007FFF` | `32 KB` | 升级接收、校验、跳转 |
| App | `0x08008000 ~ 0x0807E7FF` | `474 KB` | 主业务程序 |
| Upgrade State | `0x0807E800 ~ 0x0807EFFF` | `2 KB` | 升级状态页 |
| Diag Persist | `0x0807F000 ~ 0x0807F7FF` | `2 KB` | 诊断持久化页 |
| Config Persist | `0x0807F800 ~ 0x0807FFFF` | `2 KB` | 参数持久化页 |

说明：

- 升级时只允许擦写 App 区
- `Upgrade State / Diag / Config` 三页都必须保留

## 3. 升级链路

推荐本地升级链路：

- `维护电脑 -> USB转485 -> 现场485 -> USART3 -> STM32`

阶段切换如下：

1. 设备处于 App 运行态时，使用 `Modbus RTU`
2. 通过维护命令请求进入 Bootloader
3. 设备软件复位
4. Bootloader 启动后，切换为本手册定义的二进制升级协议

## 4. App 阶段入口命令

进入本地升级模式前，设备必须仍处于 App 运行态。

### 4.1 解锁

```text
0a060030a55a73d5
```

### 4.2 进入 Bootloader

写 `0x0031 = 0x0005`：

```text
0a0600310005197d
```

说明：

- 该命令返回成功后，设备会软件复位
- 复位后不再响应 App 阶段的 Modbus 参数维护命令
- 后续必须切换为 Bootloader 协议发包

## 5. Bootloader 通信参数

- 物理串口：`USART3`
- 485 方向控制：`PA5`
- 串口参数：`115200, 8N1`
- 协议类型：自定义二进制帧
- 单帧 `PAYLOAD` 上限：`256` 字节

## 6. Bootloader 帧格式

每帧格式如下：

| 字段 | 长度 | 说明 |
|---|---:|---|
| `SOF0` | 1 | 固定 `0x55` |
| `SOF1` | 1 | 固定 `0xAA` |
| `CMD` | 1 | 命令码 |
| `SEQ` | 1 | 序号，由上位机自增 |
| `LEN` | 2 | 负载长度，小端 |
| `PAYLOAD` | `LEN` | 负载 |
| `CRC16` | 2 | 对 `CMD+SEQ+LEN+PAYLOAD` 计算 Modbus CRC16，小端 |

说明：

- `CRC16` 算法与 Modbus RTU CRC16 一致
- `SOF` 不参与 CRC 计算

## 7. Bootloader 命令

### 7.1 `GET_INFO`

- 命令码：`0x01`
- 请求负载长度：`0`
- 响应命令码：`0x82`

响应负载结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| `boot_version` | `uint16` | Bootloader 版本 |
| `protocol_version` | `uint16` | 升级协议版本 |
| `app_base` | `uint32` | App 基地址 |
| `app_max_size` | `uint32` | App 最大尺寸 |
| `page_size` | `uint16` | Flash 页大小 |
| `state` | `uint16` | 当前升级状态 |
| `written_bytes` | `uint32` | 已写入字节数 |
| `last_error` | `uint16` | 最近错误码 |
| `reserved` | `uint16` | 保留 |

说明：

- `GET_INFO` 用于快速确认 Bootloader 是否在线
- 建议上位机或脚本启动时优先发一次 `GET_INFO`

### 7.2 `START`

- 命令码：`0x02`
- 请求负载长度：`12`

请求负载结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| `image_size` | `uint32` | 固件镜像总长度 |
| `image_crc32` | `uint32` | 整个镜像的 CRC32 |
| `target_fw_version` | `uint32` | 目标固件版本号 |

行为：

- Bootloader 先把状态切到 `ERASING`
- 擦除 App 区
- 成功后切到 `PROGRAMMING`

### 7.3 `DATA`

- 命令码：`0x03`
- 请求负载长度：`6 + chunk_len`

请求负载结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| `offset` | `uint32` | 当前包写入偏移 |
| `chunk_len` | `uint16` | 当前包数据长度 |
| `chunk_data` | `byte[]` | 当前包数据 |

限制：

- 当前要求 `offset` 必须等于 `written_bytes`
- 不接受乱序写入
- 单包 `chunk_len` 不得超过 `250`
- 原因：`DATA` 负载里还包含 `offset(4) + chunk_len(2)` 共 `6` 字节头部，因此 `6 + chunk_len <= 256`

### 7.4 `END`

- 命令码：`0x04`
- 请求负载长度：`0`

行为：

- 要求 `written_bytes == image_size`
- Bootloader 计算 App 区 CRC32
- CRC32 正确且向量表有效时，状态写为 `DONE`
- 当前实现中，`END` 成功后设备不会在该命令响应帧发完前强行切走
- Bootloader 在后续复位/重新进入启动路径时，若检测到 `DONE` 且 App 有效，则跳转到 App

### 7.5 `ABORT`

- 命令码：`0x05`
- 请求负载长度：`0`

行为：

- 中止本次升级
- 状态写为 `FAILED`
- 保持停留在 Bootloader

### 7.6 `QUERY_STATUS`

- 命令码：`0x06`
- 请求负载长度：`0`
- 响应命令码：`0x83`

响应负载结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| `state` | `uint16` | 当前升级状态 |
| `last_error` | `uint16` | 最近错误码 |
| `image_size` | `uint32` | 当前目标镜像大小 |
| `written_bytes` | `uint32` | 当前已写入字节数 |
| `last_ok_offset` | `uint32` | 最近一次成功写入后的末尾偏移 |
| `image_crc32` | `uint32` | 当前目标镜像 CRC32 |

## 8. Bootloader 响应

### 8.1 `ACK`

- 响应命令码：`0x80`
- 负载：`uint16 last_error`

### 8.2 `NACK`

- 响应命令码：`0x81`
- 负载：`uint16 error_code`

### 8.3 `INFO`

- 响应命令码：`0x82`
- 负载：见 `GET_INFO`

### 8.4 `STATUS`

- 响应命令码：`0x83`
- 负载：见 `QUERY_STATUS`

## 9. 状态定义

| 值 | 名称 | 说明 |
|---|---|---|
| `0x0000` | `IDLE` | 空闲 |
| `0x0001` | `REQUESTED` | App 已请求进入升级 |
| `0x0002` | `ERASING` | 正在擦除 App 区 |
| `0x0003` | `PROGRAMMING` | 正在写入固件 |
| `0x0004` | `VERIFYING` | 正在做最终校验 |
| `0x0005` | `DONE` | 升级完成，可跳 App |
| `0x0006` | `FAILED` | 升级失败，停留 Bootloader |

## 10. 错误码定义

| 值 | 名称 | 说明 |
|---|---|---|
| `0x0000` | `BOOT_ERR_NONE` | 无错误 |
| `0x0001` | `BOOT_ERR_BAD_FRAME` | 帧格式错误或接收超时 |
| `0x0002` | `BOOT_ERR_BAD_CMD` | 命令码不支持 |
| `0x0003` | `BOOT_ERR_BAD_LENGTH` | 长度非法 |
| `0x0004` | `BOOT_ERR_BAD_STATE` | 当前状态不允许执行该命令 |
| `0x0005` | `BOOT_ERR_BAD_SIZE` | 镜像大小越界 |
| `0x0006` | `BOOT_ERR_FLASH_ERASE` | Flash 擦除失败 |
| `0x0007` | `BOOT_ERR_FLASH_PROGRAM` | Flash 编程失败 |
| `0x0008` | `BOOT_ERR_BAD_OFFSET` | 分包偏移非法或不连续 |
| `0x0009` | `BOOT_ERR_VERIFY` | 最终校验失败 |
| `0x000A` | `BOOT_ERR_NOT_COMPLETE` | 收包未完成就请求结束 |
| `0x000B` | `BOOT_ERR_BAD_CRC` | 协议帧 CRC16 错误 |
| `0x000C` | `BOOT_ERR_TIMEOUT_RECOVERY` | 升级长时间卡住后自动恢复到 App |

## 11. 推荐升级流程

### 11.1 现场操作步骤

1. 设备正常上电并进入 App
2. 使用 Modbus RTU 解锁
3. 发送“进入 Bootloader”命令
4. 等待设备复位
5. 上位机切换到 Bootloader 二进制协议
6. 发送 `GET_INFO`
7. 发送 `START`
8. 按顺序发送 `DATA`
9. 发送 `END`
10. 收到 `ACK` 后等待设备跳转 App
11. App 启动后重新用 Modbus RTU 读取版本号确认

### 11.2 分包建议

- 建议 `chunk_len = 128` 或 `250`
- 建议每发一包都等 `ACK/NACK`
- 如果超时，先发 `QUERY_STATUS`
- 如果设备仍在 `PROGRAMMING`，按 `written_bytes` 继续续传

### 11.3 Python 脚本方式

当前仓库提供辅助脚本：

- [stm32_local_upgrade.py](/D:/Project/STM32_Mill/stm32_local_upgrade.py)

安装依赖： 如果已安装不需要再次执行

```powershell
python -m pip install pyserial
```

查看当前串口号：

```powershell
python -m serial.tools.list_ports
```

方式 A：设备已在 Bootloader   要更换COM口

```powershell
python D:\Project\STM32_Mill\stm32_local_upgrade.py run --port COM12 --chunk-size 250 --timeout 5 --verbose
```

用途：

- 发送 `GET_INFO`
- 发送 `START`
- 自动按顺序发送全部 `DATA`
- 发送 `END`
- 最后查询一次状态

注意：

- `COMx` 必须替换为电脑当前实际枚举到的 USB 转 485 串口号
- 不建议假定固定是 `COM6 / COM10 / COM12`
- 当前硬件拓扑下，`USART3` 在 App 阶段与 `G780S` 共线
- 因此前置自动切换并不总是稳定
- 更稳妥的流程仍然是：
  1. 先用维护命令让设备进入 Bootloader
  2. 再运行 `stm32_local_upgrade.py`

## 12. 当前实测结果

截至 `2026-03-23`，第二阶段本地升级已完成一次实机闭环验证，结果如下：

- App 运行态可通过 `0x0031 = 0x0005` 进入 Bootloader
- 电脑可通过 `USB转485 -> USART3` 发送 `GET_INFO / START / DATA / END / QUERY_STATUS`
- 测试镜像：`App.bin`
- 实测镜像大小：`24544 (0x00005FE0)` 字节
- 实测镜像 CRC32：`0xCFF765A7`
- `END` 后 `QUERY_STATUS` 返回：
  - `state = 0x0005 (DONE)`
  - `written_bytes = image_size = 24544`
  - `image_crc32 = 0xCFF765A7`
- 复位后 Bootloader 可识别有效 App 并跳转回 App

说明：

- `state = 0x0005` 是 `DONE`，表示升级完成，不是异常状态
- 现场若看到 App 运行时 LED 周期性停闪，当前已确认主要原因是 App 主循环中的传感器/继电器轮询为阻塞式实现，不代表升级失败
- App 侧阻塞点主要在 `USART2` 轮询链路，例如 `Temperature_Read()`、`Weight_Read()`、`Relay_ReadAllCoils()`、`Relay_ReadInputPack()`
- 若设备连续多次上电仍卡在 `REQUESTED / ERASING / PROGRAMMING / VERIFYING`，且当前 App 仍有效，当前 Bootloader 已增加“超时恢复”策略：
  - 自动把状态改为 `FAILED`
  - 错误码写为 `BOOT_ERR_TIMEOUT_RECOVERY`
  - 然后放行回 App

## 13. 中断恢复语义

当前第二阶段承诺的是：

- 升级过程中断电后，不跳坏 App
- 设备重新上电后留在 Bootloader
- 可查询状态并重新发包
- 若设备连续多次上电仍卡在升级中，且当前 App 有效，允许自动恢复到 App

当前第二阶段不承诺：

- 自动回滚到旧固件
- 双镜像无损恢复

## 14. 验收建议

建议至少完成以下验收：

- 正常升级成功
- 升级后版本号变化正确
- 升级后参数页保留
- 升级后上电次数连续
- 中途断电后停留 Bootloader
- 错包 / 坏包被拒绝
- `QUERY_STATUS` 可用于断点判断

## 15. 当前说明

本手册描述的是第二阶段当前已经落地的最小本地升级协议。

后续如果进入第三阶段远程 OTA，优先复用：

- 当前 Flash 分区
- 当前 Upgrade State
- 当前 `START/DATA/END/QUERY_STATUS` 状态机
