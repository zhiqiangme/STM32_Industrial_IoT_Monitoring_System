# STM32 Mill 第二阶段本地升级手册

本文档说明 `2026-03-25` 起 STM32 本地升级链路的实际落地方式。

当前结论：

- App 运行态仍使用 `Modbus RTU`
- 进入 Bootloader 的维护命令不变
- Bootloader 收升级包的传输协议已从“自定义二进制帧”切换为 `YMODEM`
- 当前只覆盖“本地串口升级”，不涉及远程升级链路

## 1. 适用范围

- MCU：`STM32F103ZET6 / STM32F103ZE`
- 本地升级物理通道：`USART3 + RS485`
- App 运行态维护协议：`Modbus RTU`
- Bootloader 传输协议：`YMODEM`

## 2. 当前分区

| 区域 | 地址范围 | 大小 | 说明 |
|---|---|---:|---|
| Bootloader | `0x08000000 ~ 0x08007FFF` | `32 KB` | 升级接收、校验、跳转 |
| App | `0x08008000 ~ 0x0807E7FF` | `474 KB` | 主业务程序 |
| Upgrade State | `0x0807E800 ~ 0x0807EFFF` | `2 KB` | 升级状态页 |
| Diag Persist | `0x0807F000 ~ 0x0807F7FF` | `2 KB` | 诊断持久化页 |
| Config Persist | `0x0807F800 ~ 0x0807FFFF` | `2 KB` | 参数持久化页 |

约束：

- 升级时只允许擦写 App 区
- `Upgrade State / Diag / Config` 三页必须保留

## 3. 升级链路

推荐链路：

- `维护电脑 -> USB转485 -> 现场485 -> USART3 -> STM32`

阶段切换：

1. 设备在 App 运行态时，通过 `Modbus RTU` 发送维护命令
2. App 写入升级请求状态并软件复位
3. Bootloader 启动后停留在升级态
4. Bootloader 开始通过 `YMODEM` 等待升级包
5. 维护电脑通过 `stm32_local_upgrade.py` 发送 `App.bin`

## 4. App 阶段入口命令

进入本地升级模式前，设备必须仍处于 App 运行态。

### 4.1 解锁

```text
0a060030a55a73d5
```

### 4.2 进入 Bootloader

写 `0x0031 = 0x0005`：

```text
0a0600310005197d
```

说明：

- 该命令返回成功后，设备会软件复位
- 复位后不再响应 App 运行态的业务维护寄存器
- 后续升级数据必须切换到 Bootloader 的 `YMODEM` 传输

## 5. Bootloader 通信参数

- 物理串口：`USART3`
- 485 方向控制：`PA5`
- 串口参数：`115200, 8N1`
- 传输协议：`YMODEM`

Bootloader 行为：

- 驻留后会周期性发出 `C (0x43)`，表示请求使用 CRC16 的 YMODEM 会话
- 维护电脑收到后发送 YMODEM 头包和数据包
- Bootloader 在每包写入成功后返回 `ACK`
- 整个文件结束后，Bootloader 会做 App 区 CRC32 与向量表有效性校验

## 6. 当前 YMODEM 适配说明

当前工程里的 YMODEM 不是原样照搬 ST IAP 示例，而是做了项目内适配：

- 去掉了 `flash_if / menu / common / UartHandle` 这类示例工程依赖
- 改为由 Bootloader 提供串口收发回调和 Flash 写入回调
- 仍然保留 YMODEM 的 `SOH / STX / EOT / ACK / NAK / C` 交互方式
- 保留每包 `CRC16-CCITT/XMODEM` 校验

### 6.1 头包内容

YMODEM 头包里使用：

- 文件名：`App.bin`
- 文件信息字段：`"<image_size> 0x<crc32> <target_fw_version>"`

示例：

```text
123456 0x89ABCDEF 0
```

含义：

- 第 1 个字段：镜像字节数，十进制
- 第 2 个字段：整包 CRC32，十六进制
- 第 3 个字段：目标固件版本，当前本地升级默认可填 `0`

说明：

- 第 2 个字段是当前工程额外使用的元数据，用于保留整包 CRC32 校验能力
- 第 3 个字段用于兼容现有状态页里的 `target_fw_version`
- 如果后续远程升级要复用本链路，可继续沿用这三个字段

## 7. Bootloader 内部状态机

Bootloader 仍沿用现有状态页和状态机：

| 值 | 名称 | 说明 |
|---|---|---|
| `0x0000` | `IDLE` | 空闲 |
| `0x0001` | `REQUESTED` | App 已请求进入升级 |
| `0x0002` | `ERASING` | 正在擦除 App 区 |
| `0x0003` | `PROGRAMMING` | 正在写入固件 |
| `0x0004` | `VERIFYING` | 正在做最终校验 |
| `0x0005` | `DONE` | 升级完成，可跳 App |
| `0x0006` | `FAILED` | 升级失败，停留 Bootloader |

当前错误码仍沿用：

| 值 | 名称 | 说明 |
|---|---|---|
| `0x0000` | `BOOT_ERR_NONE` | 无错误 |
| `0x0001` | `BOOT_ERR_BAD_FRAME` | YMODEM 会话异常或发送方主动中止 |
| `0x0003` | `BOOT_ERR_BAD_LENGTH` | 包长度非法 |
| `0x0004` | `BOOT_ERR_BAD_STATE` | 当前状态不允许写包 |
| `0x0005` | `BOOT_ERR_BAD_SIZE` | 镜像大小越界 |
| `0x0006` | `BOOT_ERR_FLASH_ERASE` | Flash 擦除失败 |
| `0x0007` | `BOOT_ERR_FLASH_PROGRAM` | Flash 编程失败 |
| `0x0008` | `BOOT_ERR_BAD_OFFSET` | 顺序写入偏移异常 |
| `0x0009` | `BOOT_ERR_VERIFY` | 最终 CRC32 或向量表校验失败 |
| `0x000A` | `BOOT_ERR_NOT_COMPLETE` | 收包未完成就试图结束 |
| `0x000B` | `BOOT_ERR_BAD_CRC` | 协议包 CRC16 错误或会话错误过多 |
| `0x000C` | `BOOT_ERR_TIMEOUT_RECOVERY` | 升级长时间卡住后自动恢复到 App |

## 8. 推荐升级流程

### 8.1 现场操作

1. 设备上电并确认当前 App 正常运行
2. 使用 Modbus 维护命令先解锁
3. 发送“进入 Bootloader”命令
4. 等待设备软件复位
5. 运行 `stm32_local_upgrade.py`，等待它收到 Bootloader 发出的 `C`
6. 工具通过 YMODEM 发送 `App.bin`
7. Bootloader 收完后自动做最终校验
8. 校验通过后设备自动复位并重新进入启动路径
9. 若状态为 `DONE` 且 App 有效，则 Bootloader 跳转到 App

### 8.2 Python 脚本

脚本文件：

- [stm32_local_upgrade.py](/D:/Project/STM32_Mill/stm32_local_upgrade.py)

依赖安装：

```bash
python -m pip install pyserial
```

查看串口：

```bash
python -m serial.tools.list_ports
```

设备已经在 Bootloader 时，典型命令：

```bash
python D:\Project\STM32_Mill\stm32_local_upgrade.py run --port COM12 --timeout 5 --verbose
```

如果要显式带目标版本：

```bash
python D:\Project\STM32_Mill\stm32_local_upgrade.py run --port COM12 --target-fw-version 0x010200 --timeout 5
```

说明：

- 默认镜像路径是 `STM32\MDK-ARM\Objects\App.bin`
- 脚本会自动计算整包 `CRC32`
- 脚本内部会自动选择 `128` 或 `1024` 字节的 YMODEM 数据包
- 旧版 `--chunk-size` 参数已保留兼容，但在 YMODEM 模式下不再生效

## 9. 当前已验证的保护策略

- 升级过程中只擦写 App 区
- 参数区和诊断区不参与擦除
- 中途掉电后，设备会停留在 Bootloader，允许重新完整发送
- 若设备连续多次上电仍卡在升级进行中，且当前 App 仍有效，则会自动标记为超时失败并回 App
- 升级成功后，Bootloader 会把最终 CRC32 写回状态页，供下次启动复核

## 10. 联调建议

- 串口调试时优先观察 Bootloader 是否持续发出 `C`
- 若 PC 端一直等不到 `C`，优先检查设备是否真的进入 Bootloader
- 若头包刚发出就被双 `CA` 中止，优先检查镜像大小是否超出 App 区
- 若数据发到末尾失败，优先检查 `App.bin` 是否由当前 `App` target 正常导出
- 若传输完成但设备不跳转 App，优先检查向量表是否正确以及 App 链接地址是否仍为 `0x08008000`

## 11. 本阶段边界

当前阶段已经完成的是：

- App 通过维护命令进入 Bootloader
- Bootloader 通过 `YMODEM` 接收本地升级包
- 写入完成后做整包 CRC32 和向量表校验
- 升级失败后停留 Bootloader，允许重新升级

当前阶段还没有做的是：

- 远程升级数据传输
- 双分区 A/B 镜像
- 自动回滚到旧固件

本手册只描述当前已经落地的本地升级链路。
