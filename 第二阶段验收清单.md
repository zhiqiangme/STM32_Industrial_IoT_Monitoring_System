# STM32 Mill 第二阶段验收清单

本文档用于对第二阶段“STM32 本地升级”做阶段性收口验收。

第二阶段验收目标：

- App 可通过维护命令进入 Bootloader
- Bootloader 可通过 `USART3 + 485` 完成本地整包升级
- 升级状态、保存快照、升级快照和通信异常分类可用于远程排障
- 升级后 App 可正常启动，配置区/诊断区不被破坏
- 升级中断后设备不会跳坏 App，并具备继续恢复或重新升级的能力

当前阶段不包含：

- 双镜像 A/B 回滚
- 云平台侧 OTA 下载
- 板载 RTC 绝对时间戳

说明：

- 当前诊断中的“最近一次保存时间”实现为“最近一次成功保存时的系统运行秒数快照”，不是 RTC 实时时间
- 当前 `REG_DIAG_LAST_UPGRADE_SOURCE` 记录的是最近一次 Upgrade State 中的 `request_source`
- 如果通过 `G780S` 请求进入 Bootloader，但实际升级数据由维护电脑串口发起，则成功升级后该值当前预期可能为 `0x0001 (LOCAL)`，这不是异常

## 1. 测试准备

测试前确认：

- STM32 已烧录当前第二阶段固件
- 设备 App 态可响应 G780S 远程维护命令
- 维护电脑已连接 `USB转485 -> USART3`
- Python 环境可运行 [stm32_local_upgrade.py](/D:/Project/STM32_Mill/stm32_local_upgrade.py)
- 串口工具已开启 `HEX` 模式
- 逐条发命令，等待返回后再发下一条

当前已知常用命令：

- 解锁：`0a060030a55a73d5`
- 提交保存：`0a060031000118be`
- 进入 Bootloader：`0a0600310005197d`
- 读维护区 `0x0032` 开始 6 个寄存器：`0a0300320006657c`
- 读配置区 `0x0020` 开始 9 个寄存器：`0a0300200009857d`
- 读诊断区 `0x0038` 开始 34 个寄存器：`0a03003800224565`
- 读保存/升级快照 `0x004B` 开始 15 个寄存器：`0a03004b000f74a3`
- 读升级快照 `0x004F` 开始 3 个寄存器：`0a03004f00033567`
- 读 CRC 错误计数 `0x0045` 开始 2 个寄存器：`0a0300450002d4a5`
- 读 UART ORE/FE/NE/RX_OVERFLOW 分类计数 `0x0052` 开始 8 个寄存器：`0a0300520008e4a6`
- 读 RX_OVERFLOW `0x0058` 开始 2 个寄存器：`0a030058000244a3`

Bootloader 手工帧：

- `GET_INFO(seq=1)`：`55 AA 01 01 00 00 50 18`
- `QUERY_STATUS(seq=1)`：`55 AA 06 01 00 00 51 6C`
- `ABORT(seq=1)`：`55 AA 05 01 00 00 51 28`

推荐升级命令：

```powershell
python D:\Project\STM32_Mill\stm32_local_upgrade.py run --port COM12 --chunk-size 250 --timeout 5 --verbose
```

## 3. 当前已补测通过项

以下项目已在 `2026-03-23` 完成补测并通过。

### 3.1 升级后参数区保留

目的：

- 验证升级不会破坏配置区
- 验证最近一次保存快照不会因升级被冲掉

步骤：

1. 解锁：`0a060030a55a73d5`
2. 把 `0x0020` 改成一个明确值，例如 `5000ms`
3. 提交保存：`0a060031000118be`
4. 读取维护区：`0a0300320006657c`
5. 记录：
   - 当前配置序号
   - `0x004B ~ 0x004E` 的保存快照
6. 发 `0x0031 = 0x0005` 进入 Bootloader
7. 跑一次完整升级
8. 回到 App 后读取：
   - 配置区：`0a0300200009857d`
   - 维护区：`0a0300320006657c`
   - 保存/升级快照：`0a03004b000f74a3`

预期：

- `0x0020` 仍为升级前保存值
- 配置序号不回退
- `LAST_SAVE_SEQ` 仍保持升级前最近一次保存的序号
- `LAST_SAVE_UPTIME` 不会因升级被清零

判定：

- 满足以上条件则通过

实测结论：

- 已通过
- 升级前保存 `0x0020 = 0x1388 (5000ms)`
- 升级后再读配置区，`0x0020` 仍为 `0x1388`
- 配置序号保持为 `4`
- `LAST_SAVE_SEQ = 4`
- `LAST_SAVE_UPTIME` 未因升级被清零

### 3.2 中途中断后停留 Bootloader

目的：

- 验证升级中断后不会误跳坏 App
- 验证 Bootloader 状态可查询

步骤：

1. 发送“进入 Bootloader”
2. 启动升级脚本
3. 当脚本进度达到 `20% ~ 80%` 时断电或断开设备电源
4. 重新上电，但不要先让 App 正常跑业务
5. 使用串口工具在 Bootloader 通道发送：

```text
55 AA 01 01 00 00 50 18
```

```text
55 AA 06 01 00 00 51 6C
```

预期：

- `GET_INFO` 有响应
- `QUERY_STATUS` 显示状态不是 `DONE`
- 如果中断点已超过首包写入，`written_bytes` 应大于 `0` 且小于 `image_size`
- 设备停留在 Bootloader，不会直接跳到损坏 App

判定：

- 满足以上条件则通过

说明：

- 如果中断得过早，`written_bytes` 可能仍为 `0`
- 这种情况下应重新做一次并在 `DATA` 进度明显开始后再断电

实测结论：

- 已通过
- `GET_INFO` 正常返回
- `QUERY_STATUS` 返回：
  - `state = 0x0003 (PROGRAMMING)`
  - `image_size = 0x000064E8`
  - `written_bytes = 0`
- 本次中断点偏早，因此 `written_bytes` 仍为 `0`
- 设备仍停留在 Bootloader，未误跳损坏 App

### 3.5 ORE / FE / NE 分类计数

目的：

- 验证 UART 分类计数不只是总数

步骤：

1. 先读取：

```text
0a0300520008e4a6
```

2. 分项测试：

- `FE`：把上位机串口临时改成错误波特率或错误停止位，发送少量数据后恢复 `115200 8N1`
- `ORE`：持续高速发送垃圾流，尽量制造接收来不及清空的场景
- `NE`：在台架条件允许时，通过人为引入噪声或不稳定接线制造噪声错误

3. 恢复正常串口参数后再次读取 `0x0052 ~ 0x0059`

预期：

- 对应分类计数出现增加
- 总 `UART_ERR` 计数同步增加

判定：

- 至少 `FE` 或 `ORE` 任一项被稳定触发则通过基础增强验收

说明：

- `NE` 在普通现场不容易稳定复现，可列为“台架增强项”，不是当前收口的阻塞条件

实测结论：

- 当前为部分通过
- 已确认：
  - `RX_OVERFLOW` 可稳定触发并计数
- 当前未确认：
  - `ORE`
  - `FE`
  - `NE`
- `0x0052 ~ 0x0057` 当前实测仍为 `0`
- `UART_ERR_TOTAL` 当前实测仍为 `0`
- 该项建议保留为“增强验证项”，不阻塞第二阶段收口
