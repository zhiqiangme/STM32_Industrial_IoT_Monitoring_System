# STM32 Mill 第三阶段远程升级方案

本文档用于定义第三阶段“STM32 远程升级”的可落地方案。

第三阶段不是推翻第二阶段重做，而是在第二阶段已经落地的：

- `Bootloader`
- `Upgrade State`
- `START / DATA / END / QUERY_STATUS`
- `USART3` 升级入口

基础上，把“本地维护电脑直连升级”扩展成“远端通过 `G780S` 发起和执行升级”。

适用对象：

- MCU：`STM32F103ZET6 / STM32F103ZE`
- 当前通信拓扑：
  - `USART2`：现场采集主站
  - `USART3`：远程维护 / 升级通道
  - `G780S`：远程网络透传模块

## 1. 第三阶段目标

第三阶段只解决一件事：

- 让远端能够稳定地把升级包送到现场设备，并完成一次可追踪、可恢复、可验收的远程升级

本阶段目标：

- 远端可通过 `G780S -> USART3` 发起升级
- 继续复用第二阶段 `Bootloader` 升级协议
- 支持远程查询升级状态、失败原因和最近一次升级快照
- 网络中断后可通过 `QUERY_STATUS` 判断是否续传或重传
- 升级后保留参数区、诊断区和升级状态区
- 远端可区分“进入 Bootloader 请求来源”和“升级执行结果”
- 为后续服务器侧批量升级、版本管理、任务编排打基础

本阶段不承诺：

- 双镜像 A/B 回滚
- 断网后自动后台完成升级
- G780S 本地缓存整包后离线升级
- 云平台侧完整运维系统

## 2. 当前基础

第二阶段已经提供以下能力：

- App 运行态可通过 `Modbus RTU` 请求进入 Bootloader
- Bootloader 可通过 `USART3` 接收升级包
- 已有 `GET_INFO / START / DATA / END / ABORT / QUERY_STATUS`
- `Upgrade State` 已可记录：
  - `state`
  - `request_source`
  - `image_size`
  - `image_crc32`
  - `written_bytes`
  - `last_ok_offset`
  - `error_code`
- App 诊断区已可读取：
  - 最近一次升级请求来源
  - 最近一次升级状态
  - 最近一次升级失败原因

因此第三阶段的核心不是重写 MCU 升级逻辑，而是把远端传输、会话管理和异常恢复补齐。

## 3. 当前约束

### 3.1 Flash 约束

当前仍沿用第二阶段分区：

| 区域 | 起始地址 | 结束地址 | 大小 | 说明 |
|---|---:|---:|---:|---|
| Bootloader | `0x08000000` | `0x08007FFF` | `32 KB` | 固定保留 |
| App | `0x08008000` | `0x0807E7FF` | `474 KB` | 升级目标区 |
| Upgrade State | `0x0807E800` | `0x0807EFFF` | `2 KB` | 升级状态页 |
| Diag Persist | `0x0807F000` | `0x0807F7FF` | `2 KB` | 诊断页 |
| Config Persist | `0x0807F800` | `0x0807FFFF` | `2 KB` | 参数页 |

远程升级阶段仍然只允许擦写 `App` 区。

### 3.2 通信链路约束

- `G780S` 到服务器的网络链路时延和抖动不可控
- `G780S` 现场链路本质上仍是串口透传，不适合一次发超大连续数据
- Bootloader 阶段只有 `USART3` 可用
- 当前 MCU 没有外部 Flash 暂存区
- 当前 Bootloader 仍是“单 App 区流式写入”

### 3.3 语义约束

第三阶段必须继续保持第二阶段已经验收通过的语义：

- 升级失败不能破坏参数页
- 升级失败不能破坏诊断页
- 升级中断时不能跳坏 App
- 中断后要么停在 Bootloader 等待续传，要么按超时恢复策略回 App

## 4. 推荐总体架构

推荐采用“服务器编排 + G780S 透传 + STM32 Bootloader 执行”的最小闭环方案。

### 4.1 角色边界

#### 服务器

负责：

- 保存升级包
- 计算镜像 `size / crc32 / version`
- 下发升级任务
- 按顺序发送 `START / DATA / END`
- 根据 `QUERY_STATUS` 决定续传、重传或终止
- 记录升级结果

#### G780S

负责：

- 透传远端数据到 `USART3`
- 透传 STM32 回包到服务器
- 不负责解释 Bootloader 协议内容
- 不负责在本阶段缓存整包或重组镜像

#### STM32 App

负责：

- 接受远程维护命令
- 写入 `Upgrade State.REQUESTED`
- 标记请求来源
- 软件复位进入 Bootloader
- 在升级完成后，通过诊断区暴露快照和结果

#### STM32 Bootloader

负责：

- 接收 Bootloader 协议
- 擦写 App 区
- 更新 `Upgrade State`
- 校验镜像完整性
- 成功后切回 App
- 失败后停在 Bootloader 或执行超时恢复

### 4.2 推荐数据路径

推荐路径：

- `服务器 -> G780S 网络链路 -> G780S 透传 -> USART3 -> STM32`

推荐原因：

- 复用第二阶段链路，不改 Bootloader 协议
- 现场改动最小
- 先把远程 OTA 主链路打通，再考虑后续增强

当前不建议优先做的路径：

- 先把升级包完整下发到 G780S 本地文件系统，再由 G780S 转写 STM32

原因：

- 会引入额外的文件管理、断点管理和存储约束
- 第三阶段第一版并不需要这层复杂度

### 4.3 核心链路

第三阶段第一版的核心链路可以直接理解为：

- 本地升级时，维护电脑通过串口向 `USART3` 发送 Bootloader 帧
- 远程升级时，服务器通过和 `G780S` 的网络连接发送同样的 Bootloader 帧
- `G780S` 只负责把网络侧收到的原始字节透明转发到 `USART3`
- STM32 Bootloader 仍然只认第二阶段已经落地的二进制升级协议

也就是说，第三阶段并不是另起一套 OTA 协议，而是把第二阶段已经存在的：

- `GET_INFO`
- `START`
- `DATA`
- `END`
- `ABORT`
- `QUERY_STATUS`

从“本地串口直连发送”改成“服务器通过 G780S 透传发送”。

对服务器脚本来说，核心变化只有一件事：

- 本地方案：`serial.write(frame)` / `serial.read(...)`
- 远程方案：`socket.send(frame)` / `socket.recv(...)`

升级会话本身仍然是同一套状态机、同一套分包逻辑、同一套 CRC 校验。

补充说明：

- 若现场网络拓扑是 `G780S` 主动连服务器，则服务器侧脚本通常工作在“监听”模式，等待设备连接
- 若现场网络允许电脑直接连到 `G780S` 对外开放的地址/端口，则服务器侧脚本也可以工作在“主动连接”模式
- 无论是哪种网络建链方式，只要最终建立的是稳定的原始字节双向通道，Bootloader 发包逻辑都不需要改变

## 5. 控制面与数据面

建议把第三阶段拆成两层：

### 5.1 控制面

控制面仍然走 App 运行态维护协议，负责：

- 读取版本和诊断区
- 判断是否允许升级
- 请求进入 Bootloader
- 升级完成后回读结果

### 5.2 数据面

数据面只在 Bootloader 阶段启用，负责：

- `GET_INFO`
- `START`
- `DATA`
- `END`
- `ABORT`
- `QUERY_STATUS`

这样做的好处是：

- App 阶段仍保持 Modbus 维护协议
- Bootloader 阶段继续保持简单二进制升级协议
- 两层职责清晰，排障也更容易

## 6. 第三阶段推荐流程

### 6.1 升级前检查

服务器发起升级前，建议先读取：

- 固件版本号
- 协议版本号
- 上电次数
- 最近一次升级来源 / 状态 / 错误
- 当前关键参数

目的：

- 避免对已经是目标版本的设备重复升级
- 升级前留存基线信息，便于失败后排障

### 6.2 进入 Bootloader

1. 服务器通过 G780S 发送解锁命令
2. 服务器发送 `REG_MAINT_COMMAND = ENTER_BOOT_UPGRADE`
3. App 将 `Upgrade State` 置为 `REQUESTED`
4. App 记录请求来源为远程升级
5. App 软件复位
6. Bootloader 启动并进入升级接收态

建议：

- 第三阶段新增一个明确的来源值，例如：
  - `0x0003 = REMOTE_OTA`
- 不再把真正的远程升级和普通 G780S 维护命令混为同一来源语义

### 6.3 建立 Bootloader 会话

服务器在切入 Bootloader 后应按以下顺序执行：

1. 发 `GET_INFO`
2. 发 `QUERY_STATUS`
3. 判断当前状态：
   - `REQUESTED`：可开始 `START`
   - `PROGRAMMING`：根据 `written_bytes` 决定续传或重头升级
   - `FAILED`：根据错误码决定 `ABORT` 后重启流程，或直接重发 `START`
   - `DONE`：若 App 尚未切回，优先等待或触发复位

### 6.4 实际远程发包流程

把“远程发包”拆开看，实际就是同一条网络连接上分两个阶段发两类包：

#### 阶段 A：App 运行态控制面

服务器先通过 `G780S` 当前的网络透传链路，向 App 发送现有 Modbus RTU 维护命令：

1. 发解锁命令
2. 发 `REG_MAINT_COMMAND = ENTER_BOOT_UPGRADE`
3. App 写 `Upgrade State.REQUESTED`
4. App 记录升级请求来源
5. App 软件复位

这一阶段发的是：

- Modbus RTU 原始帧

不是：

- Bootloader 二进制升级帧

#### 阶段 B：Bootloader 数据面

STM32 复位进入 Bootloader 后，服务器继续使用同一条网络连接发原始字节，但此时发包内容切换为第二阶段已经定义的 Bootloader 帧：

1. 发 `GET_INFO`
2. 发 `QUERY_STATUS`
3. 发 `START`
4. 循环发 `DATA`
5. 发 `END`
6. 如有需要，补发 `QUERY_STATUS`

这一阶段发的是：

- `55 AA + CMD/SEQ/LEN + PAYLOAD + CRC16`

也就是第二阶段本地升级脚本当前正在生成和发送的那套帧。

#### 关键理解

从 G780S 的视角看：

- 它不需要理解 Bootloader 协议
- 它也不需要理解 STM32 当前是 App 还是 Bootloader
- 它只需要继续做透明转发

从服务器脚本的视角看：

- App 阶段发送 Modbus 维护命令
- Bootloader 阶段发送 Bootloader 二进制升级帧
- 只是“发包内容”变了，底层传输仍然是同一条 TCP 字节流

因此第三阶段第一版的远程升级，完全可以理解为：

- 用 `TCP socket` 替代本地 `COM` 串口
- 用服务器脚本替代本地维护电脑脚本
- 用 `G780S` 网络透传替代 USB 转 485 直连链路

### 6.5 发送升级包

建议：

- 默认 `chunk_len = 128` 或 `250`
- 每包都等待 `ACK/NACK`
- 每发送若干包，额外发一次 `QUERY_STATUS`
- 以 `written_bytes` 为准，不以“服务器自认为发了多少”作为唯一依据

推荐策略：

- 正常网络：`250` 字节
- 网络抖动较大：切到 `128` 字节

### 6.6 完成与回切

1. 数据全部写完后发 `END`
2. Bootloader 校验整包
3. 若成功，写 `DONE`
4. Bootloader 在 `END` 响应发送完成后，建议延时短暂窗口再软件复位
5. Bootloader 重启后检测到 `DONE + App有效`
6. 跳转 App
7. App 启动后远端重新读版本号和升级快照确认结果

说明：

- 第二阶段已经允许“`DONE` 后下次启动时跳 App”
- 第三阶段建议把“成功后自动软件复位回 App”补成默认行为
- 否则远程场景下会留下“升级已完成但仍停在 Bootloader 等待人工复位”的缺口

## 7. 断点续传与恢复策略

### 7.1 当前推荐语义

第三阶段建议承诺的语义是：

- 网络中断后，设备不刷死
- 重连后可通过 `QUERY_STATUS` 判断当前进度
- 服务器可选择从 `written_bytes` 处继续续传
- 如果状态不可信，则允许重新 `START` 整包重传

### 7.2 续传判定建议

满足以下条件时，允许续传：

- `state = PROGRAMMING`
- `image_size` 与目标镜像一致
- `image_crc32` 与目标镜像一致
- `written_bytes <= image_size`
- `last_ok_offset == written_bytes`

否则建议整包重传。

### 7.3 超时恢复

继续保留第二阶段已实现的策略：

- 若设备连续多次上电仍卡在：
  - `REQUESTED`
  - `ERASING`
  - `PROGRAMMING`
  - `VERIFYING`
- 且当前 App 仍有效
- 则 Bootloader 自动把状态置为：
  - `FAILED`
  - `error_code = TIMEOUT_RECOVERY`
- 然后放行回 App

第三阶段只需要把该结果纳入远端平台判读即可。

## 8. 对现有代码的建议改动

第三阶段优先做最小改动，不建议一开始就大改 Bootloader。

### 8.1 App 侧建议

- 将“远程 OTA 请求来源”与“普通 G780S 维护请求”区分开
- 在进入 Bootloader 前写清楚：
  - `request_source`
  - `target_fw_version`
- App 启动后继续同步 Upgrade State 到诊断区

### 8.2 Bootloader 侧建议

- 保持第二阶段协议不变
- `END` 成功后增加自动软件复位
- 保留 `QUERY_STATUS`
- 保留严格顺序写入语义
- 保留 `chunk_len <= 250`

### 8.3 上位机 / 服务器侧建议

- 复用现有 [stm32_local_upgrade.py](/D:/Project/STM32_Mill/stm32_local_upgrade.py) 的协议实现
- 拆出“串口收发”和“会话编排”两层
- 在远程场景新增：
  - 重连
  - 超时重试
  - 状态查询
  - 断点续传
  - 升级日志记录

## 9. 建议分阶段落地

### 9.1 第三阶段第一版

目标：

- 远端像“虚拟维护电脑”一样，直接通过 G780S 透传 Bootloader 帧

特点：

- MCU 改动最少
- 最快拿到第一版远程 OTA 闭环
- 适合先验证真实网络环境

### 9.2 第三阶段第二版

在第一版跑稳后，再考虑增强：

- 更精细的重试窗口
- 失败后的自动重连续传
- 设备分组升级
- 升级包版本管理
- 平台任务编排

### 9.3 当前不建议优先做

- G780S 本地缓存整包
- 复杂分块重组协议
- 加密分发和签名体系
- 双镜像切换

这些都不是不能做，而是当前阶段容易把主线带偏。

## 10. 第三阶段验收建议

建议至少验收以下场景：

### 10.1 正常远程升级

- 远端请求进入 Bootloader 成功
- 远端完成一次整包升级
- App 可自动恢复运行
- 版本号正确更新

### 10.2 参数保持

- 升级前已保存参数
- 升级后参数不丢
- 保存快照不被清零

### 10.3 网络中断恢复

- 远程发包中途中断
- 重连后能读 `QUERY_STATUS`
- 可根据 `written_bytes` 继续续传或重新升级

### 10.4 超时恢复

- 设备长时间停在升级态
- 重新上电后能按策略恢复 App
- 远端可通过诊断区读到：
  - `source`
  - `state`
  - `error`

### 10.5 失败可追溯

- 失败后远端可读到最近一次升级状态和失败原因
- `CRC_ERR / RX_OVERFLOW` 等基础诊断仍可读取

### 10.6 边界保护

- 远程升级不擦 `Upgrade State`
- 不擦 `Diag Persist`
- 不擦 `Config Persist`

## 11. 当前结论

当前项目进入第三阶段时，最合适的路线不是重写一套新 OTA 协议，而是：

- 继续复用第二阶段 `Bootloader` 协议
- 继续使用 `USART3` 作为唯一升级入口
- 由服务器承担升级编排和重试逻辑
- 由 `G780S` 保持透明转发
- 由 `Upgrade State + 诊断区` 提供断点和结果判读

这条路线的优点是：

- 改动小
- 可快速拿到第一版远程 OTA 闭环
- 与第二阶段现有代码和文档连续
- 后续仍可逐步增强，而不需要推翻当前实现
