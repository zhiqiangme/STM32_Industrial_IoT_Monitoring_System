# STM32 Mill 待办事项

本文档把当前后续工作分成四类：

- 已完成：已经落地到当前代码
- 建议做：当前阶段值得推进，投入产出比高
- 可后置：有价值，但不必马上做
- 当前不建议优先做：不是不能做，而是现在做收益不高或容易把重点带偏

## 一、已完成

### 1. 远程参数维护补强

已完成内容：

- 为远程参数增加更严格的上下限检查
- 对关键 16 位参数在单寄存器写入时直接拦截非法值
- 对关键参数禁止写入 0 或超范围值
- 在提交保存前做整体一致性校验
- 细化维护错误码分类

当前已落地的组合校验：

- `flow_sample_period_ms <= sensor_period_ms`
- `di_debounce_ms < sensor_period_ms`

说明：

- 16 位参数会在写入当下校验
- 32 位参数因为拆高低寄存器写入，范围检查会在提交保存时统一校验

### 2. 只读诊断寄存器

已完成内容：

- 固件版本号
- 协议版本号
- 编译日期/时间
- 系统运行时长
- 上电次数
- 最近一次重启原因
- 最近一次非法写入地址和值
- 当前配置来源
- Modbus CRC 错误计数
- 串口异常计数
- 最近一次维护命令结果
- 最近一次非法读取地址

### 3. 第一阶段收口

已完成内容：

- 非法参数写入拦截已完成实测

- 非法读取记录已完成实测
- 参数掉电/重启保持已完成实测

## 二、建议做

### 1. 第二阶段：STM32 本地升级收口

建议按三个小步骤推进：

- 完成本地升级联调
- 完成本地升级验收记录
- 完成上位机或调试脚本侧的发包流程固化
- 补齐升级成功/失败后的诊断项回读
- 完成异常场景实测：
  - 正常升级
  - 中断升级后停留 Bootloader
  - 错包/坏包拦截
  - 参数区保留验证

建议原因：

- 第一阶段已经收口，当前主线应转入第二阶段
- 第二阶段的基础框架和最小协议已经具备，当前重点不再是继续空转设计
- 在远程 OTA 之前，必须先把“本地可升级、可恢复、可验收”闭环做完整

### 2. 远程维护诊断补强

建议优先加这些：

- 最近一次保存时间
- 最近一次保存序号的显式镜像
- 更细的通信异常分类
- 最近一次升级请求来源
- 最近一次升级失败原因

建议原因：

- 远程排障价值很高
- 不影响第一阶段收口结果
- 可以在第二阶段设计期间穿插补充

### 3. 自动/手动模式相关

- 保持现有模式框架不变
- 后续自动控制逻辑统一挂到自动模式下
- 如现场规则明确，再补自动模式下的联锁/动作参数

建议原因：

- 模式框架本身已经具备，不建议再次作为独立主线重复建设
- 后续重点应放在“业务逻辑如何挂接到自动模式”而不是“是否保留该模式”

- 自动模式下的具体联锁规则
- 复杂业务规则联锁逻辑

不建议现在优先做的原因：

- 当前现场逻辑尚未明确
- 现在写进去大概率会返工
- 应先把模式框架、参数框架和诊断能力做好

### 2. Flash 擦写次数统计

- Flash 擦写次数

不建议现在优先做的原因：

- 不是没有意义，而是当前收益不高
- 如果设计不好，反而可能增加额外写 Flash 负担
- 可以等后续统一设计持久化统计方案时再做

1. 完成第二阶段本地升级联调
2. 完成本地升级验收与异常场景验证
