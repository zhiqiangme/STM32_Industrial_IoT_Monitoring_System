# STM32 Mill 远程维护指令手册

本文档用于说明当前 App 阶段“远程参数维护”功能的寄存器定义、操作流程和常用 HEX 报文。

当前远程维护链路为：

`远端调试工具/服务器 -> G780S 网络透传 -> USART3 -> STM32 Modbus RTU 从机(地址 10)`

注意：

- 当前使用的是 `Modbus RTU` 原始帧，不是网络 AT 指令
- 发送时必须使用 `HEX` 模式
- 每条指令都必须包含正确的 `CRC16`
- 写配置区只是暂存，必须执行“提交保存”才会写入 Flash 并生效
- 若执行“进入本地升级”命令，设备会软件复位进入 Bootloader
- 进入 Bootloader 后，不再使用本文档中的 Modbus 维护命令，需改用第二阶段升级协议

## 1. 通信参数

- 从站地址：`0x0A`，十进制 `10`
- 协议：`Modbus RTU`
- 当前物理通道：`USART3`
- G780S 作用：网络透传，不负责解释 STM32 参数含义

### 3.1 当前配置寄存器实际含义

#### `0x0020 REG_CFG_SENSOR_PERIOD_MS`

- 含义：STM32 主循环里进行一轮现场采集的周期
- 影响：
  - 温度读取
  - 称重读取
  - 继电器状态读取
  - 数据刷新到 G780S 寄存器镜像

#### `0x0021 REG_CFG_FLOW_SAMPLE_MS`

- 含义：流量计换算用的采样周期
- 影响：`Flowmeter_Update()` 的计算节拍

#### `0x0022 REG_CFG_DI_DEBOUNCE_MS`

- 含义：继电器输入 DI 的去抖时间

#### `0x0023 REG_CFG_TEMP_THRESHOLD_X10`

- 含义：温度变化达到该阈值后，`PUSH_SEQ` 才递增
- 单位：`0.1℃`
- 例如：
  - 写 `10` 表示 `1.0℃`
  - 写 `25` 表示 `2.5℃`

#### `0x0024~0x0025 REG_CFG_PPL_X100`

- 含义：每升脉冲数
- 存储方式：放大 100 倍的 32 位整数
- 当前默认值：
  - `660.00`
  - 十进制存储为 `66000`
  - 十六进制为 `0x000101D0`

#### `0x0026~0x0027 REG_CFG_HZ_PER_LPM_X100`

- 含义：频率换算到 L/min 的系数
- 存储方式：放大 100 倍的 32 位整数
- 当前默认值：
  - `11.00`
  - 十进制存储为 `1100`
  - 十六进制为 `0x0000044C`

#### `0x0028 REG_CFG_CONTROL_MODE`

- 含义：控制模式
- 取值：
  - `0` = 手动模式
  - `1` = 自动模式

当前实现行为：

- 手动模式：
  - 允许本地按键控制继电器
  - 允许远程 `REG_RELAY_CTRL` 和 `REG_RELAY_CMD_BITS` 控制继电器
- 自动模式：
  - 当前暂时屏蔽本地按键和远程直接继电器控制
  - 仍保留现场采集、状态上报和参数维护能力
  - 后续自动控制逻辑将在此模式下接入

### 5.3 诊断补充说明

- `REG_DIAG_LAST_CMD_RESULT` 记录最近一次 `REG_MAINT_COMMAND` 的执行结果
- `0` 表示最近一次维护命令执行成功
- 非 `0` 时，取值与 `REG_MAINT_LAST_ERROR` 使用同一套错误码定义
- `REG_DIAG_LAST_BAD_READ_ADDR` 记录最近一次非法读取的起始地址
- `REG_DIAG_LAST_SAVE_UPTIME_H/L` 记录的是“最近一次成功保存时的系统运行秒数”，不是 RTC 实时时间
- `REG_DIAG_LAST_UPGRADE_SOURCE/STATE/ERROR` 来自 Upgrade State 页镜像，可用于查看最近一次升级请求来源和失败原因
- `REG_DIAG_UART_ERR_H/L` 是 UART 异常总数，`ORE/FE/NE/RX_OVERFLOW` 则提供更细分类
- 当前实现中，`REG_DIAG_UART_ERR_H/L` 统计的是 `ORE + FE + NE` 总数，不包含 `RX_OVERFLOW`
- `RX_OVERFLOW` 需要单独读取 `0x0058 ~ 0x0059`
- 当前实现中，如果通过 `G780S` 请求进入 Bootloader，但实际升级数据由维护电脑串口发起，则升级成功后 `REG_DIAG_LAST_UPGRADE_SOURCE` 可能表现为 `0x0001 (LOCAL)`，这符合当前实现语义；若进入 Bootloader 后发生超时恢复，则该字段可能保留为 `0x0002 (G780S)`
- `RX_OVERFLOW` 是“软件接收缓冲打满后继续收到字节”的累计次数，不等同于“发送字节数 / 缓冲长度”；现场连续发送大量相同字节时，出现“发送很多字节但只累计数次溢出”属于正常现象

### 5.4 升级请求来源定义

| 值 | 含义 |
|---|---|
| `0` | 无 |
| `1` | 本地维护电脑 |
| `2` | G780S 远程维护命令 |
| `3` | 预留远程 OTA |

### 5.5 升级状态与错误码说明

- `REG_DIAG_LAST_UPGRADE_STATE` 使用第二阶段 Bootloader 的状态定义：
  - `0=IDLE`
  - `1=REQUESTED`
  - `2=ERASING`
  - `3=PROGRAMMING`
  - `4=VERIFYING`
  - `5=DONE`
  - `6=FAILED`
- `REG_DIAG_LAST_UPGRADE_ERROR` 使用第二阶段 Bootloader 的错误码定义，详细含义请参考 [第二阶段本地升级手册](/D:/Project/STM32_Mill/第二阶段本地升级手册.md)

### 7.5 进入本地升级模式

说明：

- 该命令只负责让 App 写升级请求标志并软件复位
- 复位后设备进入 Bootloader
- 后续升级发包流程请参考第二阶段手册
- 进入 Bootloader 后，不再响应本文档中的 Modbus 维护寄存器读写
- Bootloader 升级阶段建议使用 `电脑 + USB转485 + 本地串口` 直连 `USART3`
- 第一次联调不建议在网络调试窗口中手工发送整包 `DATA`

```text
0a0600310005197d
```

## 8. 状态位定义

读取 `0x0032 REG_MAINT_STATUS`。

| 位 | 宏名 | 含义 |
|---|---|---|
| bit0 | `G780S_STATUS_UNLOCKED` | 当前处于解锁状态 |
| bit1 | `G780S_STATUS_STAGED_DIRTY` | 配置区已修改但尚未提交 |
| bit2 | `G780S_STATUS_CONFIG_LOADED` | 当前配置来自 Flash 成功加载 |
| bit3 | `G780S_STATUS_SAVE_OK` | 最近一次保存成功 |
| bit4 | `G780S_STATUS_ERROR` | 当前存在错误 |

例如：

- `0x000D = 0000 0000 0000 1101b`
- 表示：
  - 已解锁
  - 配置来自 Flash
  - 最近一次保存成功

### 9.1 当前已启用的校验规则

- `REG_CFG_SENSOR_PERIOD_MS`：`200~60000ms`
- `REG_CFG_FLOW_SAMPLE_MS`：`100~10000ms`
- `REG_CFG_DI_DEBOUNCE_MS`：`10~5000ms`
- `REG_CFG_TEMP_THRESHOLD_X10`：`1~500`
- `REG_CFG_PPL_X100`：`100~10000000`
- `REG_CFG_HZ_PER_LPM_X100`：`10~1000000`
- `REG_CFG_CONTROL_MODE`：只能写 `0` 或 `1`
- 整体组合关系：
  - `flow_sample_period_ms <= sensor_period_ms`
  - `di_debounce_ms < sensor_period_ms`

说明：

- 16 位参数在单寄存器写入时就会直接做范围拦截
- 32 位参数因为拆成高低寄存器写入，最终范围在“提交保存”时统一校验

## 10. 常用读取报文

### 10.1 读取维护状态区 `0x0032` 开始 6 个寄存器

发送：

```text
0a0300320006657c
```

示例返回（示意，末尾 CRC 以设备实际返回为准）：

```text
0a030c000d0000000100000002000992dd
```

解析：

- `0x0032 = 0x000D` 状态位
- `0x0033 = 0x0000` 最近错误码
- `0x0034 = 0x0001` 配置结构版本
- `0x0035~0x0036 = 0x00000002` 当前配置序号
- `0x0037 = 0x0009` 解锁剩余 9 秒

### 10.2 读取配置区 `0x0020` 开始 9 个寄存器

发送：

```text
0a0300200009857d
```

示例返回（示意，末尾 CRC 以设备实际返回为准）：

```text
0a0312271003e80032000a000101d00000044c0000ed20
```

解析：

- `0x0020 = 0x2710 = 10000ms`
- `0x0021 = 0x03E8 = 1000ms`
- `0x0022 = 0x0032 = 50ms`
- `0x0023 = 0x000A = 1.0℃`
- `0x0024~0x0025 = 0x000101D0 = 66000 = 660.00`
- `0x0026~0x0027 = 0x0000044C = 1100 = 11.00`
- `0x0028 = 0x0000 = MANUAL`

### 10.3 读取诊断区 `0x0038` 开始 34 个寄存器

发送：

```text
0a03003800224565
```

返回格式说明：

- `0x0038~0x0039`：固件版本、协议版本
- `0x003A~0x003C`：编译日期时间
- `0x003D~0x003E`：运行时长秒数
- `0x003F~0x0040`：上电次数
- `0x0041`：重启原因
- `0x0042~0x0043`：最近非法写入地址和值
- `0x0044`：当前配置来源
- `0x0045~0x0046`：Modbus CRC 错误次数
- `0x0047~0x0048`：UART 异常总次数
- `0x0049`：最近一次维护命令结果
- `0x004A`：最近一次非法读取起始地址
- `0x004B~0x004C`：最近一次保存时的运行秒数快照
- `0x004D~0x004E`：最近一次成功保存的配置序号
- `0x004F~0x0051`：最近一次升级来源、状态和失败原因
- `0x0052~0x0057`：UART ORE / FE / NE 分类计数
- `0x0058~0x0059`：接收缓冲溢出次数

### 10.4 读取保存/升级快照区 `0x004B` 开始 15 个寄存器

```text
0a03004b000f74a3
```

返回格式说明：

- `0x004B~0x004C`：最近一次成功保存时的运行秒数快照
- `0x004D~0x004E`：最近一次成功保存的配置序号
- `0x004F`：最近一次升级来源
- `0x0050`：最近一次升级状态
- `0x0051`：最近一次升级失败原因
- `0x0052~0x0057`：UART ORE / FE / NE 分类计数
- `0x0058~0x0059`：RX_OVERFLOW 计数

### 10.5 读取升级快照区 `0x004F` 开始 3 个寄存器

发送：

```text
0a03004f00033567
```

返回格式说明：

- `0x004F`：最近一次升级来源
- `0x0050`：最近一次升级状态
- `0x0051`：最近一次升级失败原因

### 10.6 读取通信异常分类计数

读取 `CRC_ERR`：

```text
0a0300450002d4a5
```

读取 `UART_ERR_TOTAL`：

```text
0a03004700027565
```

读取 `ORE / FE / NE / RX_OVERFLOW`：

```text
0a0300520008e4a6
```

读取 `RX_OVERFLOW`：

```text
0a030058000244a3
```

说明：

- `UART_ERR_TOTAL` 当前统计 `ORE + FE + NE`
- `RX_OVERFLOW` 不计入 `UART_ERR_TOTAL`，需单独查看
- 现场实测中，连续灌入大量无效字节后，`RX_OVERFLOW` 可能增加而 `UART_ERR_TOTAL` 仍保持不变，这属于当前实现预期

### 10.7 当前已实测的判读参考

- 保存快照正常时，`0x004B~0x004E` 应能读到非零的保存时运行秒数和最近一次保存序号
- 升级成功回到 App 后，`0x004F~0x0051` 常见表现为：
  - `0001 0005 0000`：本地串口升级完成，状态 `DONE`，无错误
- 若由 `G780S` 请求进入 Bootloader 但未完成升级，设备超时恢复到 App 后，`0x004F~0x0051` 常见表现为：
  - `0002 0006 000C`：来源 `G780S`，状态 `FAILED`，错误码为超时恢复
- `CRC_ERR` 测试时，发一帧 CRC 错误的 Modbus 报文后，`0x0045~0x0046` 应增加
- `RX_OVERFLOW` 测试时，连续灌入大量无效字节后，`0x0058~0x0059` 应增加；此时 `0x0047~0x0048` 不一定变化

## 13. 使用注意事项

- 不要把两条 RTU 帧直接拼成一条连续发送
- 每条指令发完，等返回后再发下一条
- 只有写 `0x0020~0x0028` 配置区和执行需要改配置的维护命令时才需要先解锁
- 读取数据区、配置区、维护状态区、诊断区都不需要解锁
- 改了寄存器值以后，不执行“提交保存”不会落到 Flash
- 解锁超时以后，设备会自动重新上锁
- 如果执行了“进入本地升级模式”，设备会立即离开当前 App 维护协议并复位
- 如果写寄存器返回异常帧：
  - `86 02`：通常表示地址非法
  - `86 03`：通常表示写入值非法或解锁口令错误
  - `86 04`：通常表示维护命令执行失败
  - `83 02`：通常表示读取地址非法
- 读写功能正常时，回包一般会直接回显原命令或返回 `0x03` 数据帧

## 14. 当前阶段边界

当前文档覆盖的是“App 运行态下的远程参数维护框架”，不是“Bootloader 本地升级协议”。

已经支持远程修改的是：

- 采集周期
- 流量采样周期
- DI 去抖
- 温差触发阈值
- 流量换算参数
- 自动/手动模式切换

当前还没有实现的是：

- 重量达到阈值后自动打开继电器
- 温度达到阈值后自动打开继电器
- 业务规则联锁逻辑

这些要等后续你把现场逻辑确定后，再扩展新的业务参数寄存器。

第二阶段已经新增但不在本文档中展开的是：

- 通过 `G780S_CMD_ENTER_BOOT_UPGRADE` 进入 Bootloader
- Bootloader 本地升级协议

相关说明请查看第二阶段手册。
